1.1 Domain Specific Languages in Live Coding

  • In Live Coding, the primary UI is language.

  • The Language is a central element of communication between machine, performer and audience

  • Language is a thinking tool, shaping how we perceive the art

  • A good Live Coding Language is often

    • Declarative
    • Terse
    • Abstract

  • These attributes are in contrast to the code that is required to manipulate the low level system components (speakers, monitors, etc..):

    • Imperative
    • Verbose
    • Concrete

  • When creating a Live Coding Language, the main task of the creator is to bridge that gap between abstract and concrete, between the world of ideas and the world of hardware registers.

  • A DSL is either

    • external: a custom language with its own parser.
    • embedded / internal: a language implemented within an existing host language

  • Examples:

    • Tidal: internal DSL within Haskell + External DSL for Mini Notation

1.2 Audio, Video and Language Graphs

  • In Audio Programming, the lowest abstraction is a constant stream of Numbers that describe speaker positions in time.
  • In Graphics Programming, it is a stream of numbers that describes pixel colors in time and space.
  • We can directly engage on that level, which is what Bytebeat does. Also see IBNIZ for audio visual example
  • Engaging on that level leaves us with nothing but arithmetic, which is not very declarative (but cool anyway)
  • A commonly chosen abstraction for audio are graphs, which describe the interaction of signal processing components.
  • Examples: SuperCollider, Pure Data, Max/MSP
  • Programming Languages are closely related to graphs, as the source code of a program is often translated into an Abstract Syntax Tree at compile time:
out(
  filter(
    saw(55), // sawtooth wave
    800 // filter frequency
  )
);
CallExpression: out
  CallExpression: filter
    CallExpression: saw
      Literal: 55
    Literal: 800

Trees are subsets of graphs, and conceptually, variables can turn a tree into a graph:

var frequency = 55;
out(
  filter(
    saw(frequency) + saw(frequency + 1), //
    800 //
  )
);

(see tldraw for graphic)

  • While it is still a tree on the language level, it is conceptually a graph, or more specifically, a directed acyclic graph (DAG).
  • Graphs like this are called Data Flow Graphs (DFG), which are often part of compiler optimizations.

1.3 Constructing Graphs

To turn function alls into a graph is simple:

class Node {
  constructor(type, ...ins) {
    this.type = type;
    this.ins = ins;
  }
}
const out = (input) => new Node("out", input);
const filter = (input, frequency) => new Node("filter", input, frequency);
const saw = (frequency) => new Node("saw", frequency);

const graph = out(filter(saw(55), 800));

This is the data structure we get:

{
  "type": "out",
  "ins": [
    {
      "type": "filter",
      "ins": [
        {
          "type": "saw",
          "ins": [55]
        },
        800
      ]
    }
  ]
}

1.3.1 Method Chaining

Although it is still a simple graph, the above graph defintion is already very nested. To avoid nesting, we can implement method chaining like this:

let register = (type, fn) => {
  Node.prototype[type] = function (...args) {
    return fn(this, ...args);
  };
  return fn;
};

const out = register("out", (input) => new Node("out", input));
const filter = register("filter", (input, f) => new Node("filter", input, f));
const saw = register("saw", (frequency) => new Node("saw", frequency));

const graph = saw(55).filter(800).out();

Now our graph definition is flat, following the same order as the graph visualization. The idea is simple: A Node can be constructed in 2 ways:

  1. by calling a top-level function
  2. by calling a method on an existing Node, which is then used as the first input
